Patrones de diseño en DeliGo

Documento detallado — uso real en el proyecto hasta la fecha

Proyecto: DeliGo — Sistema de delivery online (Caso de estudio 4)
Equipo / repositorio: github.com/valerybrillyt/deligo
Frontend: Angular (carpeta frontend/)
Backend: Node.js + Express (carpeta backend/)
Base de datos: SQLite — archivo database/deligo.db (visible en DBeaver)
Patrones implementados: Singleton (×3), Factory, Decorator, Composite

1. Regla fundamental del proyecto

Los cuatro patrones de diseño solicitados en clase están implementados en el backend, dentro de la carpeta backend/patterns/. El frontend no contiene la lógica de los patrones: solo consume el API REST y muestra al usuario el resultado (menú por categorías, tipos de entrega, extras, totales, etc.).

La base de datos guarda datos (usuarios, restaurantes, productos, pedidos, logs). Los patrones son código JavaScript que organiza el comportamiento del servidor.

Importante para la exposición: Si la profesora pregunta «¿dónde está el Factory?», la respuesta es: en backend/patterns/factory/orderFactory.js, y se ejecuta cuando el usuario confirma un pedido en la pantalla Menú (ruta POST /api/pedidos).

3. Singleton — «una sola instancia para todo el servidor»

El patrón Singleton garantiza que una clase tenga una única instancia compartida en toda la aplicación. En DeliGo se usa tres veces, cada una con un propósito distinto.

3.1 Singleton de base de datos

Archivo: backend/patterns/singleton/databaseConnection.js

Problema que resuelve: Si cada petición HTTP abriera su propia conexión a SQLite, el servidor sería lento y podría corromper el archivo deligo.db.

Cómo funciona:

  • La clase DatabaseConnection guarda DatabaseConnection.instance.
  • En el constructor, si ya existe instancia, devuelve la misma.
  • connect() abre database/deligo.db, crea tablas si no existen y carga datos de demo.
  • getPool().query(sql, params) es el método que usan todas las rutas del API.

Dónde se usa en el código:

  • backend/server.js — al arrancar el servidor.
  • backend/routes/api.js — en cada consulta (restaurantes, login, menú, pedidos, logs).
  • backend/patterns/singleton/logService.js — para escribir en la tabla logs.

Tablas que gestiona: usuarios, restaurantes, productos, pedidos, logs, meta.

3.2 Singleton de autenticación

Archivo: backend/patterns/singleton/authService.js

Problema que resuelve: Centralizar las sesiones de usuarios en un solo lugar del servidor.

Clase: AuthService con un Map de tokens en memoria.

Métodos principales:

  • login(userId, email) → genera token deligo_{userId}_{timestamp} y lo guarda.
  • validateToken(token) → recupera la sesión.

Flujo en la aplicación:

  1. Usuario entra a Login (/login).
  2. Frontend llama POST /api/usuarios/login con email y contraseña.
  3. El API valida en SQLite (Singleton BD) y llama authService.login() (Singleton auth).
  4. El token vuelve al frontend; se guarda en localStorage.
  5. El usuario puede acceder a Restaurantes, Menú y Seguimiento (rutas protegidas con authGuard en Angular).

3.3 Singleton de logs (auditoría)

Archivo: backend/patterns/singleton/logService.js

Problema que resuelve: Registrar de forma uniforme todo lo que hace el usuario, por si la app falla o hay que reconstruir el flujo (requisito académico de tabla de logs).

Método principal: registrar({ usuarioId, accion, detalle, ruta, ip, exito })

Tabla en BD: logs — campos: id, usuario_id, accion, detalle, ruta, ip, exito, creado_en.

Acciones que se registran automáticamente:

AcciónCuándo ocurre
LOGIN_OK / LOGIN_ERRORAl iniciar sesión
REGISTRO_OK / REGISTRO_ERRORAl crear cuenta
VER_RESTAURANTESAl listar restaurantes
VER_MENUAl abrir menú de un restaurante
CREAR_PEDIDOAl confirmar pedido (aquí intervienen Factory + Decorator)
VER_SEGUIMIENTOAl ver estado del pedido
ACCESO_PAGINA / ACCESO_DENEGADOAl navegar sin o con sesión
SESION_CERRADAAl cerrar sesión

Cómo verlo: DBeaver → tabla logs, o GET http://localhost:3000/api/logs.

4. Factory (Fábrica) — crear el tipo correcto de pedido

Archivo: backend/patterns/factory/orderFactory.js

Intención del patrón: Encapsular la creación de objetos. El código que procesa el pedido no necesita saber si es estándar, express o programado; solo llama a la fábrica.

4.1 Clases producto

Tipo (parámetro)ClaseEtiquetaTiempoEnvío
estandarPedidoEstandarEntrega normal45 min$3.50
expressPedidoExpressEntrega rápida20 min$6.00
programadoPedidoProgramadoProgramar entrega60 min$2.00

4.2 Funciones exportadas

  • crearPedido(tipo, datos) — switch que instancia la clase correcta.
  • listarTiposDisponibles() — devuelve opciones para el frontend.

4.3 Uso en frontend y API

  1. Usuario logueado abre Menú de un restaurante (ej. Pizza Express).
  2. En el panel «Tu pedido», elige radio buttons de tipo de entrega.
  3. Al cargar la página, MenuComponent llama GET /api/pedidos/tipos → Factory lista los tipos.
  4. Al pulsar «Pagar», POST /api/pedidos envía tipoPedido: 'express' (por ejemplo).
  5. En api.js: const pedido = crearPedido(tipoPedido, {...}).
  6. El pedido calcula costoEnvio y tiempoEstimadoMin según su clase.

Archivo frontend relacionado: frontend/src/app/menu/menu.component.ts — propiedad tipoPedido, método confirmarPedido().

5. Decorator (Decorador) — extras sin modificar el pedido base

Archivo: backend/patterns/decorator/orderDecorator.js

Intención del patrón: Añadir responsabilidades a un objeto dinámicamente, envolviéndolo, sin alterar las clases de la Factory.

5.1 Estructura de clases

  • PedidoBase — envoltorio inicial del pedido de la Factory.
  • DecoradorPropina — añade $2.00.
  • DecoradorSeguro — añade $1.50.
  • DecoradorEco — añade $0.75 (empaque ecológico).

Cada decorador extiende PedidoBase y delega al objeto envuelto (super.descripcion(), super.costoExtra()).

5.2 Función principal

aplicarDecoradores(pedido, extras = [])
// extras ejemplo: ['propina', 'seguro']

Recorre el array y va encadenando: Base → Propina → Seguro → …

5.3 Uso en frontend y API

  1. En Menú, sección «Complementos» con checkboxes.
  2. GET /api/pedidos/extras lista opciones disponibles.
  3. Al confirmar, el frontend envía extras: ['propina', 'eco'].
  4. En api.js: const decorado = aplicarDecoradores(pedido, extras).
  5. Total final = subtotal + envío (Factory) + decorado.costoExtra().
  6. En BD se guarda descripcion_extra con texto del decorador (ej. «Pedido express + Propina + Empaque eco»).

Archivo frontend: menu.component.ts — extrasSeleccionados, toggleExtra(), costoExtras.

Archivo: backend/patterns/composite/menuComposite.js

6.4 Uso en frontend y API

  1. Usuario entra a Menú /menu/1 (Pizza Express).
  2. GET /api/restaurantes/1/menu devuelve { productos: [...], menuArbol: [...] }.
  3. MenuComponent recibe menuArbol y en el HTML hace @for (cat of menuArbol) → muestra cada categoría y sus productos.
  4. Botón «Agregar» mete el producto en el carrito.

Archivos frontend: menu.component.html, menu.component.ts, api.service.ts método getMenu().

7. Flujo completo — un pedido de principio a fin (todos los patrones)

PASO 1 — Usuario inicia sesión
  Pantalla: Login
  Patrón: Singleton (databaseConnection + authService + logService)
  API: POST /api/usuarios/login

PASO 2 — Ve restaurantes
  Pantalla: Restaurantes (requiere sesión)
  Patrón: Singleton BD + logService
  API: GET /api/restaurantes

PASO 3 — Abre menú de Pizza Express
  Pantalla: Menú /menu/1
  Patrón: Composite (arma categorías) + Singleton BD + logService
  API: GET /api/restaurantes/1/menu

PASO 4 — Agrega pizzas al carrito, elige Express + Propina
  Pantalla: Menú (panel lateral)
  Patrón: (aún no; solo UI)

PASO 5 — Confirma pedido
  Patrón: Factory crea PedidoExpress
          Decorator añade Propina
          Singleton BD guarda en tabla pedidos
          logService registra CREAR_PEDIDO
  API: POST /api/pedidos

PASO 6 — Sigue entrega
  Pantalla: Seguimiento
  Patrón: Singleton BD + logService
  API: GET /api/pedidos/:id/seguimiento

10. Qué NO es patrón de diseño (pero complementa el proyecto)

  • authGuard / guestGuard (Angular) — protegen rutas; no son GoF, pero obligan login antes de pedir.
  • authInterceptor — envía X-Usuario-Id al API para que los logs identifiquen al usuario.
  • Tabla logs — persistencia; el patrón es logService (Singleton), no la tabla en sí.

11. Cómo demostrar los patrones en la presentación

  1. En vivo: Abrir http://localhost:4200/patrones — pantalla académica que lee el API.
  2. En código: Mostrar carpeta backend/patterns/ en VS Code.
  3. En DBeaver: Tabla pedidos (Factory+Decorator en columnas tipo y descripcion_extra) y tabla logs.
  4. En Postman o navegador: GET localhost:3000/api/patrones y GET localhost:3000/api/pedidos/tipos.
  5. Demo guiada: Login → Pizza Express → agregar pizza → Express + Propina → Pagar → Seguimiento.

12. Frases listas para decir en oral

  • Singleton: «Tenemos una sola conexión a SQLite, un solo servicio de sesiones y un solo servicio de logs para todo el servidor Node.»
  • Factory: «Cuando el cliente elige entrega express o programada, la fábrica instancia la clase de pedido correcta con su precio y tiempo.»
  • Decorator: «Los extras como propina o seguro envuelven el pedido sin modificar las clases de la fábrica.»
  • Composite: «El menú se modela como árbol: categorías que contienen productos, y el API devuelve ese árbol al frontend.»
